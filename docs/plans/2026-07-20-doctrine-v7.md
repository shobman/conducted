## The class, as the gate already detects it

Today's mechanical detection is the trailerless branch of the territory gate
(`seed/.github/workflows/gates.yml`, "Territory manifest check" step): when no `node:` trailer is
found in the PR's commit messages, the gate runs

    git diff --name-only base...head -- ':!docs' ':!CLAUDE.md' ':!work/*/brief.md' ':!work/*/*/brief.md'

and fails on any remainder. That is the **trailerless-legal** set. The v7 **bookkeeping class**
is its strict docs-only subset:

- trailerless (no `node:` trailer), AND
- every changed path matches `^docs/`, AND
- no changed path is `docs/VISION.md`, `docs/DECISIONS.md`, or `docs/BOT.md`.

Carve-out reasoning: **VISION** changes are owner-gated by existing doctrine (archival and win
clauses are owner rulings). **DECISIONS** entries are rulings — owner lane by this law's own
enumeration; today the owner's approving click on a docs PR containing a DECISIONS append IS the
countersign of the quoted ruling, and auto-merge would delete that countersign. **BOT.md** is a
measurer's configuration — the v6 authorship gate reads `mode:`/`bot-login:` from it on the base
ref; a bot that could land BOT.md changes unattended could reconfigure its own measurer (gate
integrity, applied to the new lane). Root `CLAUDE.md` and `work/*/brief.md` remain
trailerless-**legal** (v6 widening, unchanged) but owner-**lane**: law lands as owner-approved
PRs, and under bot mode the owner's required approval IS the widening ruling — auto-merging
either would delete a ruling. Consequence, for free: **doctrine migrations are excluded
mechanically** — a migration's version bump touches root `CLAUDE.md`, which fails `^docs/`; no
separate "is this a migration?" heuristic exists or is needed.

## The mechanism (honest)

GitHub blocks self-approval by the PR author, so the lane cannot be "the bot approves its own
PR." Chosen mechanism: a **`bookkeeping-merge` job in gates.yml** plus a **dedicated merge
credential only the workflow holds**.

- **Trigger `pull_request_target`:** the workflow definition and the checked-out tree are the
  BASE ref's — a PR cannot edit the lane logic it is judged by (territory-law principle applied
  to the workflow itself). The job executes NO PR content: it reads diff names, base-ref BOT.md,
  and check conclusions. (Standing warning: any future edit that checks out or executes the head
  ref inside this job reopens the classic pwn-request hole.)
- **Classification** re-derives the class from git objects — never labels, titles, or PR bodies:
  draft check → mode `bot` + `bot-login` from base-ref BOT.md → PR author == bot → no `node:`
  trailer → diff non-empty and every path in `^docs/` minus the carve-outs. Anything else exits
  green as "owner lane" (the normal case, not an error).
- **Gates-green wait:** polls the `gates` check conclusion **on the exact head SHA** (20-minute
  window; anything other than success leaves the PR for the normal path — fails closed).
- **Merge credential:** a dedicated GitHub App (suggested name `conducted-bookkeeping`;
  permissions contents:write + pull_requests:write; installed per repo), key stored ONLY as
  Actions secrets `BOOKKEEPING_APP_ID` / `BOOKKEEPING_APP_KEY`, minted per-run via
  `actions/create-github-app-token`. The owner lists the App in branch protection's
  **bypass_pull_request_allowances** — an APPROVAL-ONLY bypass. It must be a distinct principal:
  granting bypass to the bot account would hand every session holding `CONDUCTED_BOT` the
  power to merge anything green.
- **Merge:** `gh pr merge --squash --match-head-commit <verified SHA>` — a commit pushed after
  classification refuses the merge (TOCTOU pin).
- **Unconfigured repos are legal:** missing secrets → the job no-ops green and every PR stays
  owner lane. Solo-mode repos never enter the lane (mode gate).

### Every enforcement claim, pointed at its check (v6 audit discipline)

| Claim | The exact check that enforces it |
|---|---|
| Only trailerless PRs enter the lane | classify step: `git log --format=%B base..HEAD_SHA \| grep 'node: '` → owner lane |
| Lane is docs-only minus carve-outs | classify step: `ALLOW='^docs/'`, `DENY='^docs/(VISION\.md\|DECISIONS\.md\|BOT\.md)$'` over `git diff --name-only base...HEAD_SHA`; selfcheck pins both literals and tests in-lane AND out-of-lane path populations |
| A PR cannot edit its own lane logic | `pull_request_target` semantics (GitHub-defined): workflow + tree from base; the job executes no PR content |
| Doctrine migrations excluded | the version bump edits root `CLAUDE.md`, which fails `ALLOW='^docs/'` — no separate check exists or is needed |
| Only bot-authored PRs auto-land | classify step: PR author vs base-ref `bot-login:`; commit-level authorship is the v6 authorship gate inside the (required) `gates` job — both must hold |
| Merge only when gates are green | two independent layers: (a) the poll of the `gates` check conclusion on the exact head SHA; (b) GitHub server-side required status checks — bypass_pull_request_allowances waives ONLY the review requirement, so the App's `gh pr merge` is refused while required checks are red |
| No approval needed — but only for the App | branch protection still requires 1 approval for every other actor; the App is the sole bypass allowance; session tokens have no bypass (verified at wake by the v4 merge-gate check reading effective rules) |
| Nothing merges a moved target | `--match-head-commit "$HEAD_SHA"` — GitHub refuses if head moved after classification |
| Sessions cannot wield the merge credential | the App key exists only as Actions secrets, which agent sessions cannot read; `CONDUCTED_BOT` gains no new power |
| The measured party cannot reconfigure its measurers | `docs/BOT.md` in DENY; `.github/workflows/**` fails ALLOW, is never node territory, and never rides trailerless (v6 repair-path law) — workflow changes still need the owner's admin lane |

### Blast radius, stated

- By GitHub's rules the App can merge — and with contents:write, push to — anything once required
  checks pass. The ONLY thing narrowing it to class-verified PRs is this workflow's own code. That
  code's protection is the v6 repair path: `.github/workflows/**` reaches main only via the
  owner's admin lane. The lane's integrity therefore equals the integrity of main's gates.yml.
- If an owner grants the App a full-ruleset bypass instead of approval-only, the server-side
  green-checks layer disappears and only the job's own poll remains. The migration ritual forbids
  it explicitly.
- The poll finds gates by check name `gates`; a repo that renames the job breaks auto-merge
  CLOSED (no merge, owner lane behavior) — fails safe.
- The 20-minute wait window: gates slower than that leave the job red and the PR unmerged;
  a re-run or the next push retries. Fails closed.

## Dispatch-on-merge (corollary, law only)

When a bookkeeping PR merges and the dispatch board lists READY nodes, ignition may fire without
the owner — a runner or watcher launching `/conduct` on countersigned, priced briefs within the
ruled cap (the booking app already prototyped the watcher; `ops/dispatch.ps1` exists). The owner's
approval click on a showcase IS the dispatch of everything it unblocks: approval and dispatch are
one act, never two waits. v7 makes this legal; the machinery that performs it is the Desk's
`runner-ignition` node, not this law.

## Sub-agent models (owner ruling, 2026-07-20 — added to v7 scope)

Every dispatch uses the CHEAPEST model that can carry the task, and the model is named on the
dispatch itself — subagents inherit the parent's model, so an unstated model is a decision, not
a neutral default. Routing table: Principal / brief-shaping / brainstorming → frontier (fable
summoned, opus default) · conductors → opus · fleet subagents (scoped implementation, research
legs) → sonnet · mechanical sweeps (batch greps, formatting, file moves) → haiku · evaluators →
sonnet per round, frontier ONLY for the final top-cut DoD pass. Fable never does routine
implementation — frontier spend is reserved for conceptual work. Every ledger dispatch entry
records the model used (`models:` field added to `seed/templates/ledger-entry.md`). Evidence:
discovery §6.3, adversarially verified — explicit role→tier routing holds ~97% of quality at
~24% of cost, and learned routers generalize poorly to new model pools (rejected); the routing
table is exactly what conducted's role model already implies, now law instead of habit.
