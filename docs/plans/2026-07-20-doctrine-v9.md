# Doctrine v9 — two-currency budget + observed-not-queried window sensor (compact plan of record)

**Evidence, two legs — and one of them is weaker than the brief for this work assumed. Read the
honesty note before the scope.**

**(1) The owner's ruling, 2026-07-20.** Anthropic now bills Fable as paid API tokens. Two
consequences were ruled: (a) **opus is the default Principal/conductor tier; fable requires the
owner's explicit opt-in**, per invocation, for hard conceptual work only (brainstorming,
conceptualising, breaking new ideas into parts, roadmap shaping) — never routine implementation, never
long autonomous stretches; and (b) budgeting must reason about **two currencies** — PLAN-WINDOW
allowance (the 5-hour rolling + weekly limits consumed by opus/sonnet/haiku through Claude Code, an
allowance already paid for, whose exhaustion means WAIT) versus real CASH (fable API tokens and any
pay-as-you-go overflow — money out of the bank). **A node's price is a PAIR, not a number.**

**(2) The sensor constraint — evidence GAP, named.** This work was briefed to build on a field note
at the companion Desk app's `docs/field-notes/2026-07-20-window-sensor-cannot-be-headless.md`, described as the
decisive constraint. **That file does not exist** — not in the companion Desk app, not anywhere in the
estate (searched by name, by directory, and by content). Its vault's `budgets.yml` already cites
it, so the citation is live and dangling. Rather than cite a document no one can read, the drafting
session verified the underlying claims against Anthropic's published documentation:

| Claim | Verdict against published docs |
|---|---|
| The statusline stdin payload carries rate-limit data | **CONFIRMED** — `rate_limits.five_hour` and `rate_limits.seven_day`, each `{used_percentage, resets_at}`; Claude.ai subscribers only |
| It reports percentages, not quota integers | **CONFIRMED** — headroom is `100 - used_percentage` and nothing finer |
| The block appears only after the session's first API response | **CONFIRMED** — and never for API-key sessions |
| Hook payloads carry no rate-limit fields | **CONFIRMED** — the surface that DOES fire headless is blind to the window |
| The statusline does not fire headless / in `-p` mode | **UNDOCUMENTED** — statusline and headless docs are both silent; believed on an empirical probe whose write-up is missing |
| Cross-surface consumption (claude.ai/Desktop/other machines) is invisible locally | **PARTIALLY CONFIRMED, and narrower than briefed** — the `/usage` command documents that its figures come from local session history and exclude other devices and claude.ai. Whether the statusline's server-sourced `rate_limits` also excludes them is **not documented either way** |
| `/usage-credits` provides a monthly spend limit + auto-reload toggle | **CONFIRMED** — the only mechanical cash wall, and it lives in the owner's account, not this repo |

So v9 ships the sensor law with its load-bearing premise marked as a **LEAD, not a fact**, and is
deliberately architected so that being wrong about it is cheap: fail-closed-on-UNKNOWN produces
identical behaviour whether a missing reading is caused by a headless statusline or by a session that
has simply not made its first API call yet. **This is the observation rule applied to the doctrine's
own text** — see "Wording the owner should eyeball" below.

## Scope

1. **The two-currency budget law** — every cap and every ledger figure labelled `notional`
   (plan-covered) vs `cash` (real spend); pre-v9 single-number caps read as notional; fable time is
   always cash; asymmetric exhaustion (notional → WAIT and park, cash → STOP and get a ruling) and the
   explicit ban on spending cash to escape a window wait.
2. **The model-tier default**, superseding v7's routing-table default: principal/conductor → opus ·
   fleet → sonnet · sweeps → haiku · evaluator → sonnet per round, opus for the final top-cut pass ·
   **fable → owner opt-in only, per invocation, recorded in the ledger.** An unstated model is still a
   decision (subagents inherit), so the model is named on every dispatch.
3. **The window sensor law** — observed, never queried. Sessions persist
   `{value, observed_at, source, confidence}`; **absence is a distinct state, never 0**; between
   sessions the reading ages forward by deterministic `resets_at` arithmetic; a cheap probe session is
   the legal way to refresh before expensive dispatch. Bounded claims: the runner reads the window
   when Claude Code reports it and refuses expensive dispatch on a missing or stale reading — it
   **never** claims to know remaining allowance.
4. **Governor behaviour** — dispatch only while (reading FRESH AND above the reserve fraction) AND
   (cash cap has headroom); **fail CLOSED for expensive dispatch, OPEN for cheap** on UNKNOWN;
   **parked-waiting-for-window is a legal terminal state for a session, not a failure.**
5. **The hard cash stop is external** — `/usage-credits` monthly spend limit with auto-reload OFF is
   the only mechanical wall; window governance is soft, and the doctrine says so out loud.
6. **Owner assumptions are config, not inference** — reserve fraction · max reading staleness · cash
   ceiling per period · per-model weekly sub-cap policy. The last is deliberately un-hard-coded: the
   docs contradict themselves on whether the second weekly limit is Opus- or Sonnet-scoped, and the
   sensor cannot settle it (one undifferentiated `seven_day` figure, no per-model breakdown).

## Mechanical vs behavioral (v9's own enforcement-claim audit, per v8's discipline)

| Clause | Class | What actually reads it |
|---|---|---|
| Ledger/status carry `window_cost`, `cash_cost`, `models`, `window_reading`, `fable_optin` | **MECHANICAL (weak)** | The placeholder gate (`gates.yml`, `<[A-Z][A-Z0-9_-]+>`) fails an unfilled SCREAMING-CASE token. It forces the fields to be FILLED. It cannot tell whether the numbers are TRUE. |
| The `/usage-credits` monthly spend limit with auto-reload off | **MECHANICAL (real, external)** | Anthropic's billing system. The only hard cash wall in the entire law — and it is an account setting, not a line in this repo. |
| The runner's governor refusing dispatch on a stale/absent reading | **MECHANICAL where implemented, and only there** | The runner reads `budgets.yml` before dispatch. In repos with no runner this clause enforces nothing at all. Named, not implied. |
| Currency labels on caps and ledger figures | BEHAVIORAL | Nobody. No gate parses a ledger. |
| Model-tier defaults; "an unstated model is a decision" | BEHAVIORAL | Nobody. No gate inspects a dispatch's model. |
| Fable opt-in, per invocation, recorded | BEHAVIORAL | Nobody. A session can invoke fable and write `fable_optin: NONE`; only the ledger audit catches it. |
| Reserve fraction / staleness limit / cash ceiling honoured | BEHAVIORAL | Nobody. |
| Fail-CLOSED on UNKNOWN; UNKNOWN-is-never-0 | BEHAVIORAL | Nobody — except inside a runner that implements it. |
| Parked-waiting-for-window is a legal terminal state | BEHAVIORAL | Nobody; it is permission granted to sessions, which needs no enforcement. |

**Net honesty statement, stated in the law itself:** no gate reads a budget, prices a dispatch, checks
a model tier, or has ever seen a window reading. A session that dispatches an opus fleet on a stale
window against an exhausted cash cap merges exactly as green as one that parks. The two-currency law
is bookkeeping discipline plus one real external wall — saying otherwise would repeat precisely the
overclaiming that v8's audit existed to remove.

## Wording the owner should eyeball

1. **The missing field note.** The brief for this work named it the decisive constraint and it is not
   on disk. `budgets.yml` already cites it. Either it was never written, or it lives somewhere
   unsearched. **v9 cites the owner's ruling and the published docs as evidence, and cites the field
   note only as a gap.** If the note exists, the sensor clause should be upgraded from LEAD to
   confirmed and MIGRATIONS amended.
2. **Cross-surface invisibility is stated more narrowly than briefed.** The brief said cross-surface
   consumption is "structurally invisible." What is documented is that `/usage` excludes other devices
   and claude.ai; the statusline's `rate_limits` is server-sourced and the docs are silent on it. v9
   therefore says every reading is **an upper bound on headroom, never a measurement**, which is safe
   under both readings. If the owner knows the stronger claim to be true, it can be tightened.
3. **"Expensive" and "cheap" dispatch are not numerically defined.** The fail-closed/fail-open rule
   turns on that line, and v9 deliberately leaves it to the owner's ruled config rather than inventing
   a threshold. It is the most likely source of inconsistent behaviour between sessions.
4. **Notional caps as a circuit-breaker are now arguably too strict.** If the allowance is already
   paid for, a notional cap breaker stops work that costs nothing. v9 keeps it (it remains a genuine
   burn-rate signal and protects the shared window) but the owner may prefer notional caps to WARN
   rather than BREAK, with only cash caps breaking.
5. **Evaluator's final pass moved from "frontier" to opus explicitly.** Under v7 "frontier" could read
   as fable; that is now cash, so v9 names opus. Worth confirming this is the intended reading of the
   ruling rather than a downgrade of the top-cut evaluation.
